MongoDB Collections
===================

customers
├── Stores Customer documents
└── Fields: _id, name, email, phone

products
├── Stores Product documents (Polymorphic Pattern)
└── Fields: _id, name, price, category, inventory, type, warranty, brand, size, color, etc.

orders
├── Stores Order documents with embedded OrderItems (normal orders)
├── Uses Subset Pattern: customerId (reference) + customerName (subset)
├── Uses Computed Pattern: pre-calculated total
├── Uses Outlier Pattern: isLargeOrder flag for 50+ items
└── Fields: _id, customerId, customerName, orderDate, items[], total, schemaVersion, isLargeOrder, totalItemCount, bucketCount

order_item_buckets 🆕
├── Stores OrderItemBucket documents (for large orders with 100+ items)
├── Uses Outlier Pattern: splits large orders into buckets of 50 items
└── Fields: _id, orderId, bucketNumber, items[]

MongoDB's Guiding Principle
===========================

    "Data that is accessed together should be stored together."

This principle drives all data modeling decisions in this demo.


MongoDB Design Patterns Demonstrated (6 Total!)
================================================

CORE DATA MODELING PATTERNS:

1. EMBEDDING PATTERN (Data ALWAYS accessed together)
   ├── OrderItem documents embedded in Order
   ├── Why? Order items are ALWAYS accessed with the order
   ├── Result: Zero joins, single query retrieval
   └── Benefits: 100x faster than relational joins, atomic updates

2. SUBSET PATTERN (Data FREQUENTLY accessed together) ⭐
   ├── Order stores customerId (reference) + customerName (subset)
   ├── OrderItem stores productId (reference) + name/price (subset)
   ├── Why? Customer name FREQUENTLY accessed, full data RARELY needed
   ├── Result: 90% of queries need no lookup
   └── Benefits: optimized reads, historical accuracy, flexibility

3. REFERENCING PATTERN (Data OCCASIONALLY accessed together)
   ├── Order references Customer by customerId
   ├── OrderItem references Product by productId
   ├── Why? Full customer/product data OCCASIONALLY needed
   ├── Result: Can lookup complete data when needed
   └── Benefits: single source of truth, data integrity, optimal storage

SCHEMA DESIGN PATTERNS (Official MongoDB Patterns):

4. COMPUTED PATTERN 🧮
   ├── Pre-calculate and store values in documents
   ├── Example: Order 'total' calculated from items
   ├── Why? Avoid expensive calculations on every read
   └── Benefits: Fast queries, consistent results, simple code

5. POLYMORPHIC PATTERN 🎭
   ├── Different document types in same collection
   ├── Example: Electronics, Clothing, Books in 'products' collection
   ├── Why? Flexible schema, easy to add new types
   └── Benefits: Single collection, type-specific fields, no schema changes

6. DOCUMENT VERSIONING PATTERN 📋
   ├── Track schema evolution over time
   ├── Example: schemaVersion field in Order (v1 → v2)
   ├── Why? Safe schema changes, gradual migration
   └── Benefits: Backward compatibility, audit trail, no downtime

Build & Run Commands
====================

Start MongoDB:
$ docker-compose up -d

Build Project:
$ mvn clean install

Run Application:
$ mvn spring-boot:run

Run Demo:
$ ./demo-commands.sh

View in MongoDB:
$ mongosh
> use sample_pc_oms
> db.customers.find().pretty()
> db.products.find().pretty()
